# Experiment 2
要证明“我们提出的这个抽象，能同时解释并覆盖现有几类方法，而这些现有方法只是它的特例或局部版本”

也就是看这个抽象，能不能把现有几类 adaptive workflow 放进同一个框架里解释

比如：

Comparison

+ **Self-RAG** 主要解决：什么时候检索 
+ **CRAG** 主要解决：检索结果不靠谱怎么办 
+ **FLARE** 主要解决：生成过程中什么时候再检索
+ Our Method 



+ 2Wiki
+ MuSiQue
+ Bamboogle

**但，这些看起来不同，其实都可以被看成 information acquisition workflow 的不同特例。**

也就是它们都在回答同一个更一般的问题里的某一部分：

+  要不要继续获取信息？ 
+  当前信息够不够？ 
+  当前信息可不可信？ 
+  继续获取信息值不值？ 
+  什么时候该停？ 

如果能把这些方法统一写成在这个坐标系下的不同简化版本，

state / uncertainty / action / stop / cost

  
“统一的不确定性感知信息获取 workflow”是一个更一般的问题抽象



比如可以设计几个 failure mode：

1. **不该检却检了**
2. **该检却没检**
3. **检到了但没识别出证据不可靠**
4. **本该继续验证，却提前停了**
5. **本该停，却继续过搜**

然后看：

+  Self-RAG 能不能处理 1 / 2 
+  CRAG 能不能处理 3 
+  search optimization 能不能处理 5 
+  咱们Our，能不能在一个统一 workflow 下同时处理这些 

如果结果是：

+  现有方法各自在一两个 failure mode 上有效 
+  咱们的方法在多个 failure mode 上都更稳

通过一些指标，来得到一个结论，我们统一的不确定性感知信息获取 workflow，不只是一个更大的说法，而是一个能带来更高决策质量的更一般抽象









# Experiment 3
**Search workflow 把核心问题看成“如何更好地搜”；你们要把核心问题看成“在不确定下，是否还值得继续获取信息，以及该以什么形式获取信息”。** Search-R1、R1-Searcher 这类工作都把重点放在多轮搜索交互、查询生成和搜索能力强化上；HiPRAG 进一步把 over-search / under-search 作为搜索行为优化对象。它们已经非常接近 agentic retrieval，但主语仍然是 _search behavior_

__

在这里，跟“搜索行为 workflow”比

比如：

Comparision

+  HiPRAG：关注 over-search / under-search，优化搜索行为。
+  各类 RL-based / agentic search 方法：把搜索建成多步行为策略

Benchmark

+ DocBench

这类方法已经很接近咱们，但是我们要强调，不是“我们也做多步 workflow”。

他们主要优化的是 search behavior；  
咱们要定义的是 under uncertainty 的 information acquisition

这意味着我们的核心动作不只是 search，还包括：

+  verify 
+  stop 
+  abstain 
+  decide evidence sufficiency 

这是我们希望和search workflow相区分的边界，基于有些样本，最优动作不是“继续搜”，而是“停”或者“验证”。换句话说我们要聊的点！

如果结果是这样：

+  在“应该停”的样本里，search workflow 还在继续搜 
+  在“应该继续搜”的样本里，naive RAG 直接答了 
+  在“应该验证”的样本里，search workflow 不会验证，只会继续搜 
+  而你们在三类样本上都更接近正确动作 



所以我们做的是，把 search workflow 和咱们的 uncertainty-aware information acquisition workflow 放到同一个环境里，比较它们在不同 failure modes 下的决策质量
